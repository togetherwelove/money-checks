# 영수증 OCR 도입 기록

최종 갱신: 2026-04-02
상태: 계획 확정

## 목적

모델 학습 없이 한국어 영수증을 인식해 가계부 초안을 자동 작성한다.

핵심 목표:

- 사용자가 영수증 사진 1장만 올려도 지출 초안이 채워진다.
- 자동 저장하지 않고, 사용자 확인 후 저장한다.
- 기존 가계부 입력 흐름과 자연스럽게 연결한다.
- 사용량 기반 외부 OCR 과금 없이 자체 실행형 구조로 운영한다.

## 현재 프로젝트 기준

이미 준비된 요소:

- 모바일 앱에 카메라와 앨범 선택 라이브러리가 포함되어 있다.
- `ledger_entries.source_type`에 `receipt`가 이미 정의되어 있다.
- `receipt_files` 테이블이 이미 존재한다.
- Supabase 기반 인증과 저장 흐름이 이미 있다.

즉 신규 작업의 핵심은 OCR 실행 서비스, 결과 정규화, 입력 화면 연결이다.

## 기술안

### 1순위 선택

`PaddleOCR`를 사용한다.

선정 이유:

- 한국어 인식 모델을 사용할 수 있다.
- 사용량 기반 SaaS 비용이 없다.
- 영수증 OCR 실험과 후처리를 우리가 통제할 수 있다.

### 구조

1. 앱에서 영수증 촬영 또는 이미지 선택
2. 이미지를 Supabase Storage에 업로드
3. OCR 전용 Python API가 이미지를 읽어 PaddleOCR 수행
4. OCR 텍스트와 박스 정보를 프로젝트 공통 스키마로 정규화
5. 앱 입력 폼에 초안 반영
6. 사용자가 수정 후 저장
7. `ledger_entries.source_type = 'receipt'`로 저장

### 실행 위치

OCR은 `Supabase Edge Function`이 아니라 별도 Python 서비스에서 수행한다.

이유:

- PaddleOCR는 Python 런타임과 모델 파일이 필요하다.
- Edge Function에 직접 넣기보다 별도 서비스가 구현과 운영이 단순하다.
- 이후 OCR 엔진 교체도 서비스 경계 안에서 처리할 수 있다.

### 선택 보류안

- NAVER CLOVA OCR
- Google Document AI Expense Parser
- AWS Textract AnalyzeExpense
- OpenAI 비전 기반 직접 추출

보류 이유:

- 현재 우선순위는 비용 없는 자체 실행형 구조다.
- 한국어 영수증 특화 정확도는 후처리 보강으로 대응한다.

## 기능 정의

### 사용자 기능

- 입력 화면에서 `영수증 스캔` 진입
- 카메라 촬영 또는 사진 선택
- OCR 분석 진행 상태 표시
- 분석 완료 후 자동 작성된 가계부 초안 확인
- 날짜, 금액, 카테고리, 메모 수정 후 저장

### 1차 지원 범위

- 한국어 인쇄 영수증
- 이미지 1장
- 지출 항목만 지원
- 사용자 확인 후 저장 방식

### 1차 추출 필드

- 상호명
- 거래일시
- 총액
- 통화
- 카테고리 힌트
- 메모 후보
- 품목 목록(참고용)

### 1차 제외 범위

- 완전 자동 저장
- 다중 페이지 PDF
- 여러 장 동시 업로드
- 오프라인 모바일 온디바이스 OCR
- 자체 학습 모델

## 저장 구조 계획

### 기존 테이블 재사용

- `public.receipt_files`
- `public.ledger_entries`

### 신규 테이블 추가

`public.receipt_extractions`

예상 컬럼:

- `id`
- `receipt_file_id`
- `user_id`
- `book_id`
- `provider`
- `status`
- `raw_result jsonb`
- `parsed_result jsonb`
- `error_message`
- `created_at`
- `updated_at`

### 데이터 흐름

1. 이미지 업로드 후 `receipt_files` 생성
2. OCR 호출 결과를 `receipt_extractions`에 저장
3. 사용자가 저장 확정 시 `ledger_entries` 생성
4. `metadata` 또는 참조 키로 OCR 결과와 연결

## API 계획

### OCR 서비스

이름 제안: `receipt-ocr`

구현 제안:

- `FastAPI`
- `PaddleOCR`
- 필요 시 이미지 전처리 모듈 분리

역할:

- 사용자 인증 토큰 검증 또는 내부 호출 검증
- 업로드된 파일 접근 확인
- PaddleOCR 호출
- 결과 정규화
- 추출 결과 저장
- 앱에 표준 응답 반환

### 표준 응답 스키마

```json
{
  "merchant_name": "스타벅스 강남점",
  "occurred_on": "2026-04-02",
  "total_amount": 6200,
  "currency": "KRW",
  "category_hint": "식비",
  "note": "아메리카노 1잔",
  "items": [
    {
      "name": "아메리카노",
      "amount": 6200,
      "quantity": 1
    }
  ],
  "confidence": 0.93
}
```

### 정규화 원칙

- OCR 원본 결과와 정규화 결과를 분리 저장한다.
- 총액은 `합계`, `총액`, `받을금액`, `결제금액` 후보 중 우선순위로 판정한다.
- 날짜는 한국 영수증 표현 패턴을 우선 인식한다.
- 상호명은 상단 블록과 사업자 정보 주변 텍스트를 우선 사용한다.

## 운영 및 보안 원칙

- OCR 호출은 앱에서 외부 OCR 서비스 키를 직접 다루지 않는다.
- OCR 서비스는 서버 환경변수로만 구성한다.
- 원본 이미지는 사용자 본인만 접근 가능해야 한다.
- OCR 실패 시에도 직접 입력으로 전환 가능해야 한다.
- OCR 원본 응답은 디버깅에 필요하므로 보관하되, 사용자 노출은 정규화 결과만 사용한다.

## 우선 준비할 것

- PaddleOCR를 실행할 서버 또는 로컬 실행 환경
- Python 버전과 의존성 설치 방법 확정
- 한국어 영수증 샘플 10~20장
- Supabase 영수증 업로드 버킷 준비

## 리스크

- 영수증마다 총액 후보가 여러 개 나올 수 있다.
- 할인, 부가세, 봉사료, 합계가 섞여 총액 판정이 흔들릴 수 있다.
- 흐린 사진, 회전 사진, 긴 영수증은 정확도가 떨어질 수 있다.
- 자체 운영 방식이라 모델 품질 개선은 후처리와 전처리에 의존한다.
- 서버 메모리와 초기 로딩 시간이 SaaS 방식보다 무거울 수 있다.
